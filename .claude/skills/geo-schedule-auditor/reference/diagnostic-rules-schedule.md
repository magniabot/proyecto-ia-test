# Geo-Schedule Auditor — Schedule & Device Diagnostics (GS-D06–D09)

**Threshold source:** Diagnostic Thresholds Reference (REF_69), Schedule & Device section

---

## GS-D06: Device CPA Variance
**Severity:** High (10 pts) | **Data:** `device-performance.csv`

**Rule:** A device type with CPA 50%+ above the account average is significantly underperforming and warrants bid adjustment or exclusion.

**How to check:**
1. Read `device-performance.csv` → aggregate by `segments.device` (DESKTOP, MOBILE, TABLET)
2. Calculate account-level average CPA (total cost / total conversions across all devices)
3. For each device with 100+ clicks, compute deviation: `(device_CPA - avg_CPA) / avg_CPA * 100`
4. FAIL if any device CPA is 50%+ above average

**Pass/Fail:**
- PASS: All device CPAs within 50% of account average
- FAIL: Device CPA 50%+ above average — list device, CPA, deviation, and click volume
- SKIP: Any device type with <100 clicks → "Insufficient device data — need 100+ clicks per device"

**Details format:** "[DEVICE] CPA: $X (XX% above avg $Y) on Z clicks. Waste: $W"

**Waste calculation:** `waste = (device_CPA - avg_CPA) * device_conversions`

**Routing on FAIL:** `/geo-schedule-optimizer schedule` to apply device bid modifiers

## GS-D07: Ad Schedule Waste (Dead Time Windows)
**Severity:** High (10 pts) | **Data:** `schedule-performance.csv` (from `schedule-performance.gaql`)

**Rule:** Time slots with zero conversions over 4+ weeks are dead windows that should be removed via custom schedule coverage. Time slots with CPA 60%+ above average (but still converting) are high-CPA windows that should get bid modifier reductions (GS-E04). Cross-references WD-D10: dead time window without schedule exclusion.

**How to check:**
1. Read `schedule-performance.csv` → aggregate by `segments.day_of_week` + `segments.hour`
2. Group adjacent hours into 4-hour blocks for statistical significance (e.g., Mon 00:00-03:59, Mon 04:00-07:59)
3. Calculate account-level average CPA across all time slots
4. For each block with 50+ total clicks:
   - **Dead window:** Zero conversions across 4+ weeks of the data period → FAIL
   - **High-CPA window:** CPA 60%+ above average (or ROAS 60%+ below) → FAIL
5. Cross-reference with `campaign-criteria.csv` → check if dead windows are already excluded (campaign has schedule coverage that omits these hours)
6. Only flag windows that are NOT already excluded by schedule coverage

**Pass/Fail:**
- PASS: No dead or high-CPA time windows (above thresholds)
- FAIL: Dead time windows or high-CPA windows found — list each with day, hours, clicks, CPA, waste
- SKIP: Fewer than 50 clicks per time block → "Insufficient schedule data — group into broader time blocks"

**Details format:** "[DAY] [HH:00-HH:59]: 0 conversions over X weeks, $Y spent. Already excluded: no"

**Routing on FAIL:** `/geo-schedule-optimizer schedule` to pause dead time windows

## GS-D08: Schedule Consistency Validation
**Severity:** Low (3 pts) | **Data:** `schedule-consistency.csv` (from `analyze-schedule-consistency.js`)

**Rule:** Schedule patterns must be confirmed across 4+ weeks before acting on them. A single bad week is noise, not signal. This check validates whether patterns detected in GS-D07 are consistent.

**How to check:**
1. Read `schedule-consistency.csv` → generated by `analyze-schedule-consistency.js`
2. If the file contains `# insufficient_data` or fewer than 4 weeks observed → SKIP
3. For each time slot flagged in GS-D07:
   - Check `consistency` column: `confirmed` vs `unconfirmed` vs `insufficient`
   - If `unconfirmed`: WARN — the pattern may be noise
   - If `confirmed`: validate GS-D07 findings (supports the FAIL verdict)
4. Also flag any time slots with `confirmed` high/low patterns NOT caught by GS-D07

**Pass/Fail:**
- PASS: All GS-D07 findings are confirmed, no unconfirmed patterns
- WARN: Unconfirmed patterns found — flag for re-audit in 2 weeks
- SKIP: <4 weeks of data → "Need 4+ weeks to confirm patterns"

**Details format:** "X time slots with unconfirmed patterns (observed Y weeks). Re-audit in 2 weeks to confirm."

**Routing on WARN:** "Re-run `/geo-schedule-auditor schedule` in 2 weeks to confirm pattern before acting"

## GS-D09: Bid Modifier Stacking
**Severity:** Medium (5 pts) | **Data:** `campaign-criteria.csv`

**Rule:** When multiple modifier types (location + device + schedule) are active on the same campaign, their effects multiply. A +30% location modifier combined with a +30% device modifier creates a +69% effective modifier — potentially far from intended. Flag when combined effect exceeds +50% or -50%.

**How to check:**
1. Read `campaign-criteria.csv` → group by `campaign.name`
2. For each campaign, collect all active (non-zero, non-removed) bid modifiers by type:
   - Location modifiers (`campaign_criterion.type = LOCATION`)
   - Device modifiers (`campaign_criterion.type = DEVICE`)
   - Ad schedule modifiers (`campaign_criterion.type = AD_SCHEDULE`)
   - Demographic modifiers (`campaign_criterion.type = AGE_RANGE`, `GENDER`, `INCOME_RANGE`)
3. Count distinct modifier types with non-zero values
4. If 2+ modifier types active, calculate worst-case combined effect:
   - Combined = `(1 + mod_A) * (1 + mod_B) * ... - 1`
   - Example: +30% location * +30% device = `1.3 * 1.3 - 1 = +69%`
5. WARN if any campaign has combined modifier effect >+50% or <-50%

**Pass/Fail:**
- PASS: No campaigns with combined modifier effect exceeding +/-50%
- WARN: Campaigns with stacked modifiers creating unintended amplification — list campaigns with modifier breakdown and combined effect
- SKIP: No campaigns have 2+ modifier types active → "No modifier stacking detected"

**Details format:** "[Campaign]: location +X% × device +Y% × schedule +Z% = combined +W% (threshold: +/-50%)"

**Routing on WARN:** `/geo-schedule-optimizer` to review and flatten stacked modifiers
